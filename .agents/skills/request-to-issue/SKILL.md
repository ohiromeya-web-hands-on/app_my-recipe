# request-to-issue

ユーザーの曖昧な要望を、実装可能な粒度の GitHub Issue に整えて起票するための skill。誤起票を避けるため、必ずプレビューとユーザー承認を挟む。

## 前提

- 起票先は現在のリポジトリの `origin` のみ。
- GitHub 操作には `gh` CLI を使う。未認証または未インストールなら、起票前にユーザーへセットアップを依頼する。
- Issue 以外の作成、他リポジトリへの起票、既存 Issue 本文の自動編集、ラベル・アサイン・マイルストーンの自動付与は行わない。

## Workflow

1. 要望を理解する
   - ユーザーの要望を一度短く要約する。
   - 曖昧な点だけ 1〜3 問で確認する。十分具体的なら質問せず進める。
   - 必要なら関連ファイルや既存実装を読み、実装可能性と粒度を確認する。
   - すでに実装済み、または明らかにスコープ外と判断した場合は、起票前にその理由を伝え、続行可否を確認する。

2. 類似 Open Issue を確認する
   - `gh issue list --state open --limit 100` で Open Issue の全体を確認する。
   - 要望から機能名、対象ファイル名、ドメイン語などのキーワードを 2〜3 個抽出し、`gh issue list --state open --search "<keyword>"` を実行する。
   - Open Issue が limit に達している場合は、一覧だけで判断せずキーワード検索を追加して取りこぼしを防ぐ。
   - 類似 Issue が見つかった場合は番号とタイトルを提示し、ユーザーに「新規起票する / 既存 Issue にコメント追記する / 中止する」を確認する。
   - 既存 Issue にコメント追記する場合も、コメント案をプレビューして承認を得てから `gh issue comment` する。

3. Issue タイトルと本文を作る
   - タイトルは変更対象と目的が分かる長さにする。日本語でよい。
   - 下記テンプレートを基本に、該当しないセクションは省略してよい。
   - 受け入れ条件はチェックリスト形式にする。
   - 実装メモには関連ファイル、制約、参考 URL、既知の注意点を簡潔に入れる。

```markdown
## 背景 / Why

...

## 概要 / What

...

## 要件

- ...

## スコープ外

- ...

## 受け入れ条件

- [ ] ...

## 実装メモ / 補足

- ...

## 関連 Issue / PR

- ...
```

4. プレビューして承認を得る
   - タイトル案と本文案をユーザーに提示する。
   - ユーザーに「起票する / 修正する / 中止する」のいずれかを確認する。
   - 承認前に `gh issue create` を実行しない。

5. 起票する
   - 承認後、エスケープ事故を避けるため本文をファイル書き込みツールまたはエージェントのファイル書き込み機能で一時ファイルに保存してから起票する。
   - シェルの heredoc、`echo`、インライン `--body` は、日本語、コードブロック、引用符、`*` を含む本文で壊れやすいので使わない。
   - コマンド例:

```bash
gh issue create --title "<title>" --body-file /tmp/request-to-issue-body.md
```

   - `gh issue create` には、インライン `--body` ではなく必ず `--body-file` を使う。
   - 起票後、Issue URL をユーザーへ返す。

## 品質基準

- Issue タイトルは一読して変更対象と目的が分かる長さにする。
- 要件は実装者がそのまま着手できる粒度にする。
- 重複チェックを省略しない。
- ユーザー承認を省略しない。
- 起票後に、作成した Issue URL と必要なら次アクションを短く伝える。
