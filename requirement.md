# MyKitchen 要件定義書

> Notion の「マイレシピ」テンプレートを Web アプリとして再実装するにあたり、Notion が暗黙に提供している機能（認証、自動保存、リレーションの双方向同期、ブロックエディタ、共有URL、モバイル対応、バックアップ等）を自前で実装する必要がある。本書ではその差分も含めて明文化する。

**凡例**：各機能に以下のマーカーを付与する。

- 【コア】… MVP で必須（Phase 0–3 で実装完了）
- 【拡張】… Phase 4 以降、または将来バージョン

---

## 1.1 システム概要

| 項目 | 内容 |
|------|------|
| プロダクト名 | MyKitchen |
| 種別 | シングルユーザー向けレシピ管理 Web アプリ |
| 元ネタ | Notion の「マイレシピ」テンプレート |
| ゴール | 自作レシピを登録・検索・閲覧でき、必要な材料を買い物リストと連動管理できる。モバイル／PC 問わず同一体験で利用できる |
| 対象ユーザー | 料理を日常的に作る個人（家庭内シェア等は対象外） |
| 想定利用シーン | (1) デスクで新規レシピ登録 / (2) スーパーでスマホから買い物リスト確認 / (3) 調理中にタブレット／スマホで手順参照 |

#### A-2. 一覧画面に必要なビュー

| # | ビュー | 実装 | スコープ |

| 2 | お気に入り | `isFavorite = true` のフィルタービュー、ギャラリー | 【コア】 |
| 3 | データベース | 全項目テーブル、列ソート・絞り込み可、列幅はローカル記憶 | 【コア】 |
| 4 | カテゴリー別／ジャンル別サブビュー | 和食 / 洋食 / 中華 / スイーツ / 肉料理 / 魚料理 / 副菜 / 時短 / 作り置き / ヘルシー / おもてなし | 【コア】 |

#### A-3. レシピ CRUD

**書込方針（設計原則）**：

> 多フィールド同時更新（新規作成・編集フォーム）は **明示保存（Submit）**、単一フィールドのトグル系（`isFavorite`、`ShoppingItem.purchased` 等）は **オプティミスティック更新**（UI 即時反映 → 失敗時ロールバック＋エラートースト）を採用する。Notion は全自動保存だが、Web のフォーム UI では保存責務を可視化する方が安全であり、トグル系はマイクロインタラクションを優先する。

| 操作 | 要件 | スコープ |
|------|------|---------|
| 新規作成 | `/recipes/new` フルページ。未保存変更がある状態で離脱しようとした際は `beforeunload` で確認 | 【コア】 |
| 詳細閲覧 | 一覧からの遷移は **モーダル**（Next.js Intercepting Routes）、URL 直打ちは **フルページ** | 【コア】 |
| 編集 | フォーム編集＋明示保存（インライン編集は採用しない）。保存成功／失敗はトースト表示 | 【コア】 |
| 削除 | 確認ダイアログ必須 → **ソフトデリート**（`deletedAt` に現在時刻を設定）。Undo トーストを 5 秒表示、クリックで `deletedAt = null` に復元。物理削除は日次バッチ（Vercel Cron）で `deletedAt < now() - 30 days` を対象に実行 | 【コア】 |
| 競合検知（楽観ロック） | 編集フォームの hidden field に `updatedAt` を埋め込み、送信時にサーバーで現行値と比較。不一致なら `409 Conflict` を返し、最新データを表示した差分確認ダイアログを出す | 【コア】 |

> **ソフトデリート採用の理由**：Undo と物理削除の責務を分離し、ブラウザ閉じ／ナビゲーション発生で Undo が失われても復元可能にする。【拡張】の AuditLog とも整合を取りやすく、JSON エクスポート時に `deletedAt IS NULL` フィルタで自然に除外できる。

#### A-4. 検索・フィルター

- **部分一致検索**【コア】：レシピ名・メモ本文を対象。API サーバーサイドで Prisma `contains` 実装
- **フィルター**【コア】：カテゴリー（単一）／ジャンル（複数 AND）／食事タイプ（複数 AND）／難易度（範囲）／お気に入り
- **ソート**【コア】：保存日 asc/desc、難易度 asc/desc、レシピ名 asc/desc
- **複合状態保存**【コア】：URL クエリパラメータに永続化、ブラウザ戻る／進むで復元

### B. 買い物リスト管理（ShoppingList）

#### B-1. データ項目

| 項目 | 型 | 補足 |
|------|-----|------|
| 品名 | `string` (1–40字) | 表示用の名前。表記ゆれを許容するが、同一正規化名での新規作成は不可 |
| 正規化名 | `string` | 表示用の品名とは別に保持。前後空白を削除し、英数字の大文字・小文字や全角・半角の違いをできるだけそろえた比較用キー |
| ステータス | enum | `KENTOUCHU`（検討中）/ `HISSU`（必須）。`purchased` と**独立**（購入後も「必須だった」記録を保持）|
| カテゴリー | enum | 野菜・果物 / 乳製品・卵・豆腐 / 肉・魚 / 調味料 / その他 |
| 関連レシピ | `Recipe` のリレーション（多対多） | レシピの「必要な材料」から自動で双方向構築 |
| 購入済みフラグ | `bool` | |
| 作成日 | `date` | 自動 |

> `ShoppingItem` は、同じ材料が表記ゆれで重複登録されるのを防ぐため、表示用の `品名` と比較用の `正規化名` を分けて管理する。MVP では同じ `正規化名` を持つ `ShoppingItem` の新規作成は不可とし、買い物リスト内で同じ材料が増えすぎないようにする。

#### B-2. ビュー

「買うもの」（`purchased = false`）と「購入済み」（`purchased = true`）の 2 タブをテーブル表示で提供。チェック切替はワンクリックで 5 秒 Undo 可能（ソフトデリートではなく単純フラグ反転のため、Undo はクライアント側キューで保持）。

#### B-3. レシピ連携

- レシピ編集時、`IngredientPicker` で既存 `ShoppingItem` を候補表示し、選択による再利用を優先する。候補にない場合のみ新規作成（インライン）できるが、同じ正規化名の `ShoppingItem` は作成不可
- 買い物リスト側から関連レシピ名を参照可（クリックで詳細モーダル）
- **「このレシピの不足材料を買い物リストに戻す」** ボタン【コア】：レシピ紐付き `ShoppingItem` のうち `purchased = true` のものを `purchased = false`、`status = HISSU` に一括更新
- **孤立 `ShoppingItem`**（レシピ紐付きゼロ）は**自動削除しない**（他レシピで再利用される想定、手動削除のみ）

### C. ナビゲーション・共通 UI

- **ヘッダー**【コア】：固定カバー画像＋タイトル（MVP は差替不可、差替は【拡張】）
- **サイドバー**【コア】：
  - アクション：「新しいレシピを追加」「お買い物リストに追加」
  - アコーディオン：「カテゴリー別で探す」「ジャンルで探す」→ クリックで `/recipes?category=...` 等へ遷移
  - フッター：テーマ切替トグル（`light` / `dark` / `system`）
- **ダッシュボード `/`**【コア】：最新レシピ 6 件（ギャラリー）＋ 買い物リスト未購入 Top 10。2 カラム、モバイルは縦積み
- **グローバル検索**【コア】：`⌘K` / `Ctrl+K` でコマンドパレット起動（レシピ名・材料名を横断検索）

### E. 認証・セキュリティ

シングルユーザー前提だが、デモ URL を公開する以上「誰でも書き込める」状態は不可。

| 項目 | MVP 方針 | スコープ |
|------|---------|---------|
| 読取 | 全員可（レシピは公開閲覧可）。SEO・OGP の恩恵を受ける | 【コア】 |
| 書込・削除 | **Google OAuth のみ**。`Auth.js` (NextAuth) の Google Provider を採用。環境変数 `OWNER_GOOGLE_EMAILS` に許可する Google アカウントのメールアドレスをカンマ区切りで保存し、**サインイン時に email 一致検証 → 一致以外はエラー**。MVP ではオーナー本人と予備アドレスの 2 件を許可する | 【コア】 |
| セッション | JWT（HttpOnly、SameSite=Lax、Secure） | 【コア】 |
| CSRF | `Auth.js` 組込の CSRF トークン、Server Actions の組込対策 | 【コア】 |
| レート制限 | API ルートに IP 単位 60 req/min（書込系のみ）。`@upstash/ratelimit` + Upstash Redis Free tier | 【拡張】 |
| 監査ログ | 書込系操作は `AuditLog` テーブルに記録（`actor` / `action` / `targetType` / `targetId` / `at` / `ip`） | 【拡張】 |
| 秘密情報 | `.env.local` に格納（`.env.example` をコミット）。本番は Vercel 環境変数。Google OAuth 用に `GOOGLE_CLIENT_ID`、`GOOGLE_CLIENT_SECRET`、`OWNER_GOOGLE_EMAILS` を使用 | 【コア】 |

> **Google OAuth 採用の理由**：`Credentials Provider` + bcrypt は Auth.js 公式が本番非推奨で、パスワードリセット／再発行の自前実装が必要になる。Google OAuth と許可メールアドレスの照合であれば実装量が小さく、漏洩リスクも低い。料理・買い物用途の個人アプリとしても、普段使いの Google アカウントで認証できる方が自然である。

> **公開画像の前提**：レシピ詳細ページは公開閲覧可能なため、料理写真も公開 URL で配信できる画像として扱う。本アプリでは秘匿性の高い画像や、認証がないと見られない画像は扱わない。この前提により、レシピ詳細画面・一覧サムネイル・OGP 画像などで同じ画像をシンプルに利用できるようにする。

### F. データ入出力・バックアップ

Notion からの移行、および将来的なデータロスト防止のため、入出力を設計に含める。

| 項目 | 要件 | スコープ |
|------|------|---------|
| エクスポート | JSON 一括ダウンロード（`GET /api/export`）。全レシピ（`deletedAt IS NULL` のみ）＋**全 `ShoppingItem`（孤立項目含む、孤立の場合 `recipes` フィールドは空配列）**＋画像 URL リスト。認証必須 | 【コア】 |
| インポート | JSON インポート UI。スキーマ検証（Zod）、重複名の扱いを `skip` / `overwrite` / `rename` から選択 | 【拡張】 |
| Notion CSV 取込 | Notion の CSV／Markdown エクスポートをマッピングするスクリプト `scripts/import-notion.ts` | 【拡張】 |
| 自動バックアップ | Vercel Cron で毎日 1 回、エクスポート JSON を Vercel Blob に保存、世代管理 30 日 | 【拡張】 |
| 復元 | 管理画面から任意の世代バックアップを選択して復元 | 【拡張】 |

> **前提プラン**：本アプリは **Vercel Hobby プラン** で運用可能な範囲で設計する。Hobby の制約（Cron は日次 1 回まで、Production のみ実行、サーバー実行時間 10 秒以内）を超える機能は【拡張】扱い。

### 1.3.1 スコープ定義

| 区分 | 要件 |

### 1.3.3 技術的実装方針

| 区分 | 要件 |
|------|------|
| テーマ | `light` / `dark` / `system` の 3 モード。`next-themes` 使用、ユーザー選択は `localStorage` に永続 |
| バリデーション | クライアント（React Hook Form + Zod）とサーバー（API 入口で同じ Zod スキーマを再適用）の二段構え |
| エラーハンドリング | API は以下の Discriminated Union を返却する。<br>```ts<br>type ApiResult<T> =<br>  \| { ok: true; data: T }<br>  \| { ok: false; error: { code: ErrorCode; message: string; details?: unknown } };<br>```<br>`ErrorCode` は enum 化（`VALIDATION_ERROR` / `NOT_FOUND` / `CONFLICT` / `UNAUTHORIZED` / `RATE_LIMITED` / `INTERNAL` 等） |
| 楽観ロック | 更新系 API は**編集フォームの hidden field で `updatedAt` を送信**し、サーバーで現行値と一致するか検証。不一致なら `409 CONFLICT` を返す（`If-Match` ヘッダ方式は App Router で配線コストが高いため採用しない）。シングルユーザー前提で他タブ競合は稀だが、実装の保険として最小コストで入れる |
| ログ | サーバー：`pino`（structured JSON）。本番は Vercel Logs、エラーは Sentry（Free tier） |
| 監視 | Vercel Analytics（Web Vitals）+ Sentry（エラー） |
| 観測可能性 | リクエスト ID をヘッダ／ログに伝播、エラーレスポンスにも付与 |

### 1.3.4 永続化方針の切り分け（状態の置き場所ルール）

将来の実装判断をブレさせないための共通ルール。

| 状態の種類 | 保存先 | 例 |
|------------|--------|-----|
| 共有・復元可能な画面状態 | **URL クエリパラメータ** | 検索キーワード、フィルター、ソート、現在のビュー、タブ切替 |
| ユーザー個人設定 | **`localStorage`** | テーマ（light/dark/system）、テーブルの列幅、最後に開いたタブのデフォルト |
| 永続データ（全ユーザー共通の真実源） | **サーバー DB** | レシピ、買い物リスト、材料リレーション、AuditLog（【拡張】） |
| セッション内のみ有効な一時状態 | **React state (memory)** | 調理中の材料チェックリスト、モーダル開閉、ドラッグ中の並び順 |

## 1.4 Notion 機能 → Web 実装マッピング

| Notion の機能 | Web 実装での対応 | 判断理由 |
|---------------|-----------------|---------|
| アカウント・ワークスペース | `Auth.js` + Google OAuth（単一オーナー、1.2.E） | シングルオーナー前提なら OAuth が最短距離 |
| ブロックエディタ（メモ） | Markdown + `react-markdown` + プレビュー切替 | ブロックエディタ自作は過剰、Markdown で十分 |
| プロパティ（リレーション） | Prisma 中間テーブル + サーバー側トランザクションで双方向整合 | DB の制約で整合性を担保 |
| ロールアップ | サーバーで集計してレスポンスに含める（例：レシピの「紐付き `ShoppingItem` 数」） | クライアント集計は N+1 の温床 |
| ビュー保存（Favorite / Gallery / Table） | URL クエリパラメータ、ブックマーク可能 | ユーザーが URL を貼れる＝共有可能性を維持 |
| カバー画像・絵文字 | 絵文字は `emoji-mart`、カバーは MVP 固定画像 | ユーザー差替は Phase 4 以降 |
| 自動保存 | 表示系は即時反映、**フォームは明示保存**（トグル系のみ楽観更新） | ユーザー期待値と整合、誤保存事故を防ぐ |
| 共有リンク / 公開ページ | レシピ詳細は公開可（1.2.E）。MVP は固定画像または料理写真 URL を `og:image` に設定し、OGP 画像自動生成は【拡張】 | SEO の恩恵＋ポートフォリオ露出向上 |
| モバイルアプリ | PWA でホーム画面追加、モバイルファーストの UI | ネイティブ開発は工数過剰 |
| バージョン履歴 | MVP 範囲外。【拡張】の `AuditLog` で操作ログのみ保持 | diff UI は専用設計が要、スコープ外 |
| `⌘K` クイック検索 | コマンドパレット（`cmdk` ライブラリ） | |
| ドラッグで並べ替え | `dnd-kit`（キーボード対応） | a11y 要件を満たす DnD ライブラリ |
| Notion のバックアップ | JSON エクスポートは【コア】。インポートと Vercel Cron 日次バックアップは【拡張】 | |

---

## 1.5 スコープ外（MVP で実装しないもの）

- マルチユーザー・共有（シングルオーナー前提）
- リアルタイム同期（複数デバイス同時編集の衝突解決は楽観ロックで検知するのみ、マージは手動）
- レシピのバージョン履歴・差分表示
- コメント・レビュー機能
- 栄養素計算・調理時間自動計算
- AI によるレシピ提案（将来拡張として README に記載）
- ネイティブモバイルアプリ（PWA で代替）
- 画像の複数枚登録・ギャラリー
- カレンダー / 献立プランナー
- **ハード削除 UI**（ソフトデリート採用のため、物理削除はバックグラウンドジョブのみ）

## 1.6 用語集

| 用語 | 定義 |
|------|------|
| レシピ (Recipe) | 1 つの料理の情報単位。基本情報・材料・作り方・メモを持つ |
| 買い物リスト項目 (ShoppingItem) | 食材・調味料など、買い物の単位。複数レシピから参照される |
| 材料リレーション (RecipeIngredient) | `Recipe` と `ShoppingItem` の中間テーブル。分量メモを保持 |
| ビュー (View) | 一覧画面の表示切替単位（ギャラリー／テーブル／フィルター済） |
| オーナー (Owner) | 本アプリを運用する単一ユーザー。書込操作に Google OAuth 認証が必要 |
| ソフトデリート | `deletedAt` 列で「論理的に削除」し、バッチで物理削除する方式。Undo と整合 |
| 楽観ロック | 更新前後で `updatedAt` を比較し競合検知する方式。ロック取得なし |
| 【コア】／【拡張】 | 本書の機能マーカー。【コア】は MVP 必須、【拡張】は Phase 4 以降 |
