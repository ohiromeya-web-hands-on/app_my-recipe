# post-ai-tip

AIコーディングを効率化するTipsをSlackに投稿するskill。

## 目的

レビュワーまたは開発者が開発中に気づいたTipsをチームに共有する。

- 必須ではないが、知っていると便利
- 手隙や休憩時にサクッと共有できる粒度
- 評価軸付きで、読み手が導入判断しやすい形式

## 投稿フォーマット

各Tipは以下の構造で投稿する。**評価は全軸「星が多いほど良い」に統一**する。

| 項目 | 内容 |
|---|---|
| **タイトル** | 一行で何のTipsか |
| **便利さ** ⭐1〜5 | 実際の作業がどれだけ楽になるか（5=劇的に楽） |
| **導入のしやすさ** ⭐1〜5 | 導入の手軽さ（5=コマンド一発で済む、1=複数ツール連携や手作業多い） |
| **コンテキストコスパ** ⭐1〜5 | Claude/Codexのコンテキスト消費効率（5=少ないトークンで大きな効果） |
| **できること** | そのTipで何が可能になるか |
| **準備方法** | 導入手順（コマンド、設定、リンクなど） |
| **備考** | 注意点・前提・出典など（任意、なければ省略） |

⭐は5枠固定で点灯/消灯を区別する（例: 評価3 = `⭐⭐⭐☆☆`）。スケールが一目でわかるよう、必ず☆も含めて5文字で表記する。

## 手順

### 1. ユーザから一言依頼を受け取る

例: 「VSCodeのCodex拡張についてTips投稿して」「gh cliでcodexからPR作る話を投稿」

依頼内容が曖昧で調査の方向性が決まらない場合のみ、`AskUserQuestion` で1〜2問だけ確認する。基本は自走で調査する。

### 2. 調査してドラフトを作成

ユーザの一言依頼から、エージェント自身が以下を調査・記入する:

- タイトル（一行）
- 便利さ・導入のしやすさ・コンテキストコスパの3軸の評価値（⭐1〜5）
- できること
- 準備方法（実際に動くコマンド・手順）
- 備考（前提、注意点、出典URLなど）

評価値の目安:

- **便利さ**: 「毎回助かる」=5、「たまに助かる」=3、「知ってると得」=1
- **導入のしやすさ**: 「ワンコマンド/拡張インストールのみ」=5、「設定ファイル1つ編集」=3〜4、「複数ツール連携や認証設定」=1〜2
- **コンテキストコスパ**: 「skill/hookで自動化、トークン消費なし」=5、「都度プロンプトに書くがコンパクト」=3、「毎回大量の文脈を渡す必要」=1

### 3. 【必須】サブエージェントによる事実関係レビュー

プレビューをユーザに見せる**前**に、必ず `general-purpose` サブエージェントを起動して事実確認を行う。**このステップはスキップ不可**。

サブエージェントへのプロンプトには以下を含める:

- ドラフト全文（タイトル、評価、できること、準備方法、備考）
- 確認してほしい項目:
  - **コマンドや手順は実在し、記載通りに動くか**（hallucinationが特に起きやすい）
  - **ツール名・拡張機能名・URLは正確か**
  - **「できること」の主張に誇張や事実誤認はないか**
  - **前提条件（OS、必要権限、依存ツール）の漏れはないか**
- 出力形式: 各項目について「✅ 検証OK」「⚠️ 要修正（理由・修正案）」「❓ 検証不能（理由）」のいずれかで返す

サブエージェントの返答を受けて:

- ⚠️ があればドラフトを修正
- ❓ が重要項目に残る場合はユーザに確認を求めるか、備考に「未検証」と明記
- **重要項目（コマンド・URL・ツール名・「できること」の主張）に修正が入った場合は、事実関係レビューを再度実施する**（誤り→修正→新たな誤り、を防ぐため）
- ✅ のみになったら次へ

### 4. プレビューと承認

修正済みドラフトをBlock Kit形式のJSONに組み立て、ユーザに**Slackで実際に見える形**でマークダウンプレビューを示す。`AskUserQuestion` で「投稿する/編集する/中止」を確認。

備考が空なら備考セクションは出力しない。

### 5. 投稿

承認されたら、ペイロードを `/tmp/slack-tip-payload.json` に書き出して投稿する（インラインJSONはエスケープ事故が起きやすいため）。

**ペイロード生成は必ず Write tool を使う**。シェルヒアドキュメントや `echo` は `*`・日本語・引用符でエスケープ事故が起きやすいので使わない。

投稿コマンド:

```bash
REPO_ROOT=$(git rev-parse --show-toplevel)
SLACK_TIPS_WEBHOOK=$(grep '^SLACK_TIPS_WEBHOOK=' "$REPO_ROOT/.env.local" | cut -d '=' -f2-)
# .env.local が引用符付き (例: SLACK_TIPS_WEBHOOK="https://...") の場合に備えて前後の引用符を剥ぐ
SLACK_TIPS_WEBHOOK="${SLACK_TIPS_WEBHOOK%\"}"; SLACK_TIPS_WEBHOOK="${SLACK_TIPS_WEBHOOK#\"}"
SLACK_TIPS_WEBHOOK="${SLACK_TIPS_WEBHOOK%\'}"; SLACK_TIPS_WEBHOOK="${SLACK_TIPS_WEBHOOK#\'}"
[ -z "$SLACK_TIPS_WEBHOOK" ] && { echo "SLACK_TIPS_WEBHOOK が .env.local に未設定"; exit 1; }
curl -sS -X POST -H 'Content-Type: application/json' \
     --data @/tmp/slack-tip-payload.json \
     "$SLACK_TIPS_WEBHOOK"
```

- `git rev-parse --show-toplevel` でリポジトリルートを解決することで、サブディレクトリから呼ばれても確実に `.env.local` を見つけられる
- 値の前後の `"` / `'` を bash parameter expansion で剥がす。これがないと `.env.example` の `SLACK_TIPS_WEBHOOK=""` をコピーした空値が `""`（2文字）として残り空チェックをすり抜けたり、引用符付き URL が curl に literal で渡って `curl: (3)` で失敗する
- URL未設定時は無音失敗ではなく明示エラーで止める

レスポンスが `ok` であることを確認してユーザに報告。失敗時はSlackのエラーメッセージをそのまま見せる。

### 6. 後始末

`/tmp/slack-tip-payload.json` は残してOK（次回上書きされる）。

## 例

### 例1: VSCodeのCodex拡張機能

- タイトル: VSCodeのCodex拡張でGUI操作
- 便利さ: ⭐⭐⭐⭐☆
- 導入のしやすさ: ⭐⭐⭐⭐☆
- コンテキストコスパ: ⭐⭐⭐☆☆
- できること: CLIではなくVSCodeのGUI上でCodexを操作できる。差分プレビューやファイル参照が視覚的に楽
- 準備方法: VSCode拡張から「Codex」をインストール、APIキー設定
- 備考: CLI慣れしている人にはCLIの方が速い場合もある

### 例2: gh CLIでGitHub操作を委譲

- タイトル: gh CLIをinstallしてCodexからGitHubを操作
- 便利さ: ⭐⭐⭐⭐⭐
- 導入のしやすさ: ⭐⭐⭐⭐⭐
- コンテキストコスパ: ⭐⭐⭐⭐☆
- できること: CodexからPR作成・コメント取得・Issue操作などが可能になる
- 準備方法: `brew install gh && gh auth login`
- 備考: 初回の `gh auth login` のみ手動が必要

## 注意

- Webhook URLは `.env.local` の `SLACK_TIPS_WEBHOOK` から取得する（`.gitignore` 済み）
- ペイロードは一時ファイル経由で投稿（シェルエスケープ事故防止）
- 投稿成功（`ok` レスポンス）まで確認してから完了報告
- **事実関係レビュー（手順3）は省略しない**。Tipsの信頼性が低いとチャンネルの価値が下がる
- 同じTipの連投を避けるため、迷ったら投稿前にユーザに「過去投稿との重複はないか」を一言確認する
